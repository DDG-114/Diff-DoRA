# LLM Electricity Price Forecasting Plan

## Task

Use `2025` Shaanxi electricity spot-market data to train and evaluate an LLM-based day-ahead price forecasting system for a `96`-step horizon.

## Why direct 96-step LLM generation is weak

Current repo evidence already shows the failure mode:

- `h16 rolling` LLM tends to collapse to floor-price outputs or nearly constant short chunks.
- `h96` direct generation parses but error grows sharply in later steps.

## Literature direction

The papers below support a more realistic LLM route:

1. `Time-LLM`
   - Reprograms time series into an LLM-compatible format instead of naive numeric generation.
   - Link: https://arxiv.org/abs/2310.01728

2. `Large Language Models Are Zero-Shot Time Series Forecasters`
   - Shows LLMs can forecast time series, but numeric tokenization and alignment effects matter.
   - Link: https://arxiv.org/abs/2310.07820

3. `Forecasting Time Series with LLMs via Patch-Based Prompting and Decomposition`
   - Supports patch-based prompting, decomposition, and similarity-based neighbor augmentation.
   - Link: https://arxiv.org/abs/2506.12953

4. `FLAIRR-TS -- Forecasting LLM-Agents with Iterative Refinement and Retrieval for Time Series`
   - Supports iterative prompt refinement informed by retrieved analogs.
   - Link: https://arxiv.org/abs/2508.19279

5. `Day-Ahead Electricity Price Forecasting for Volatile Markets Using Foundation Models with Regularization Strategy`
   - Electricity-price-specific evidence that foundation/time-series models perform better when spike behavior is handled explicitly.
   - Link: https://arxiv.org/abs/2602.05430

6. `Regression Models Meet Foundation Models: A Hybrid-AI Approach to Practical Electricity Price Forecasting`
   - Supports hybrid pipelines where foundation-model outputs are used as enriched features for a downstream predictor.
   - Link: https://arxiv.org/abs/2603.06726

7. `Prompting Underestimates LLM Capability for Time Series Classification`
   - Relevant caution: prompt outputs can understate what the model knows; direct prompt-only generation is not always the right measurement.
   - Link: https://arxiv.org/abs/2601.03464

8. `Set the Clock: Temporal Alignment of Pretrained Language Models`
   - Relevant caution: LLMs can mix outdated temporal knowledge; retrieval and explicit temporal grounding are important for time-sensitive prediction tasks.
   - Link: https://arxiv.org/abs/2402.16797

## Repo-aligned LLM strategy

The most practical path in this repo is:

1. Predict short chunks, not the whole day directly
   - `horizon=16`
   - roll `6` times to cover `96` points

2. Keep retrieval
   - retrieve similar windows and expose their future curves

3. Add a candidate day-level skeleton curve
   - generated by the strongest non-LLM baseline
   - injected into prompts as a baseline to refine

4. Train the LLM to refine the candidate instead of generating from scratch
   - this reduces long-horizon drift
   - this is consistent with hybrid / retrieval-augmented forecasting ideas

5. Prefer residual refinement over absolute-value generation
   - let the LLM predict the correction to a candidate curve
   - this is more stable than asking it to regenerate all price levels directly
   - residual targets should be clipped / normalized during early-stage training to avoid scale explosion

## Current experimental status

### Strongest non-LLM baselines

- Tree baseline:
  - `mean_daily_mean_accuracy = 0.8123`
- Tree + sequence fused:
  - `mean_relative_mape_accuracy = 0.6373`
  - `mean_daily_mean_accuracy = 0.7542`

These are useful as candidate skeleton generators for the LLM route.

### LLM route

- Candidate/refinement code path is implemented.
- Residual-refinement path is implemented and runs stably after fixing:
  - candidate / target normalization mismatch
  - residual target clipping
  - prompt / supervision semantic mismatch
- Current `10`-day rolling result is stable but still underperforms the candidate baseline on average.

## Current diagnosis

The most important practical finding so far is:

- letting the LLM rewrite the whole short-horizon chunk is still too aggressive
- on the current `10`-day probe, the candidate baseline remains stronger on average

This suggests the next LLM step should be:

1. `selective refinement`
   - only revise steps / chunks where the candidate is likely unreliable
2. `confidence-aware refinement`
   - use uncertainty / retrieval disagreement / regime heuristics to trigger LLM edits
3. `residual-only supervision`
   - do not ask the LLM to regenerate already-good parts of the curve

## Recommended experiment matrix

### Phase A: smoke and stability

1. `LLM-h16` without candidate
2. `LLM-h16` with candidate skeleton
3. compare:
   - parse success
   - mean prediction range
   - daily mean accuracy
   - rolling 96-step MAE / RMSE

### Phase B: stronger LLM supervision

1. keep candidate skeleton in prompt
2. optionally supervise residual instead of absolute curve
3. keep retrieval and diff features

### Phase C: final comparison

1. non-LLM baseline
2. LLM-h16 rolling
3. LLM-h16 candidate-refinement rolling

## Practical expectation

Given current evidence, the most realistic path to the target is:

- use LLM as a refinement model
- not as a pure 96-step generator
- keep short-horizon generation
- inject candidate skeleton + retrieved future examples + power-system covariates

That is the strategy currently being implemented in this repo.
